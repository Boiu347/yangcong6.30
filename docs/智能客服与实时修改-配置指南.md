# 智能客服 & 实时修改 —— 克隆后配置指南

> 面向：克隆本仓库后发现「智能客服」和「实时修改」两个功能用不了的同事。
>
> **一句话结论**：这两个功能失效，99% 是因为克隆后**没有 `.env`**。`.env` 被 `.gitignore` 忽略、不会随代码提交，需要你自己创建并填写。
> - 智能客服要能出「AI 实时总结」→ 需要 `AI_API_KEY`
> - 实时修改要能「保存成功」并刷新后仍在 → 需要 `DATABASE_URL`（Postgres）

---

## 0. 名词对应（代码里的实际叫法）

| 你说的功能 | 代码里的实现 | 关键文件 |
|------------|--------------|----------|
| 智能客服 | 站内问答助手 `SiteAssistantWidget` | `client/src/components/siteAssistant/`、`server/modules/ai/` |
| 实时修改 | 编辑模式 + 内容持久化（保存到 Postgres，30s 轮询同步） | `client/src/hooks/useContentStore.ts`、`server/modules/content/`、`server/modules/research/` |

## 1. 最快跑通（TL;DR）

```bash
# 1. Node 版本必须 >= 22
node -v

# 2. 安装依赖（用 npm，仓库锁文件是 package-lock.json）
npm install

# 3. 创建并填写 .env（关键！克隆后没有这个文件）
cp .env.example .env

# 4. 启动开发环境（前端 5173 + 后端 3000）
npm run dev
# 浏览器打开 http://localhost:5173
```

`.env` 至少要填这些才能让两个功能都正常：

```bash
# 智能客服（AI 实时总结）
AI_GATEWAY_URL=https://ops-ai-gateway.yc345.tv/v1
AI_API_KEY=<向管理员索取真实 key>
AI_MODEL=claude-sonnet-4-6

# 实时修改（保存到数据库）
DATABASE_URL=postgresql://user:pass@host:5432/dbname

# 鉴权（本地自测可留空；见第 4 节的坑）
READ_PASSWORD=
EDIT_PASSWORD=
```

> 数据库表会在服务首次启动时自动创建（`CREATE TABLE IF NOT EXISTS`），**不需要**手动跑 migration 或导入 SQL。

## 2. 为什么克隆后这两个功能不行？（原理）

### 关键事实

1. **`.env` 不在 git 里**。`.gitignore` 明确忽略了 `.env`：

```
# Environment files
.env
.env.local
```

所以你 `git clone` 下来只有 `.env.example`（模板），没有真实配置。缺配置时：

2. **智能客服**：后端 `AiService` 读不到 `AI_API_KEY`（默认空字符串）时，**不会调用 AI 网关**，直接降级为「仅展示检索到的站内证据」，前端顶部标签会是 `· 仅证据检索` 或 `· AI不可用`，而不是 `· AI实时总结`。

3. **实时修改**：后端 `content` / `research` 模块读不到 `DATABASE_URL` 时，所有写接口返回 **503**，前端保存会 toast「保存失败，数据库可能未配置」。页面还能看（用代码里的默认数据），但改了刷新就没了。

### 请求链路（帮助定位问题在前端还是后端）

```
智能客服：
  前端 SiteAssistantWidget
    → 本地检索站内知识库（不依赖后端）
    → POST /api/ai/ask-site        （dev 下 Vite 代理到 :3000）
      → AiController.askSite
        → AiService.answerSiteQuestion
          → POST {AI_GATEWAY_URL}/chat/completions   （需要 AI_API_KEY）

实时修改：
  前端「进入编辑」→ POST /api/auth/verify-editor（校验 EDIT_PASSWORD）
  前端保存 → PUT /api/content/:key（带 x-access-password 头）
    → ContentController → Postgres（需要 DATABASE_URL）
```

## 3. 智能客服（站内问答助手）配置

### 依赖的环境变量

| 变量 | 作用 | 不配置时 |
|------|------|----------|
| `AI_API_KEY` | AI 网关鉴权 key | UI 可用、能出证据链接，但**没有 AI 总结**（降级 `unavailable`/`evidence`） |
| `AI_GATEWAY_URL` | AI 网关地址，默认 `https://ops-ai-gateway.yc345.tv/v1` | 有 key 但地址不可达时，AI 请求失败 → 只展示证据 |
| `AI_MODEL` | 使用的模型，如 `claude-sonnet-4-6` | 用默认值；换成推理模型（如 `claude-sonnet-5`）代码已自动兼容 |

### 验证是否成功

1. 打开页面右下角橙色浮动按钮。
2. 问一句：`为什么初中的家庭会买家庭包？`
3. 期望：回答顶部出现 **`InsightHub · AI实时总结`**，下方有可点击的证据链接。
   - 如果显示 `· 仅证据检索` 或 `· AI不可用` → `AI_API_KEY` 没配好，或网关不可达。

### 常见问题

- **一直显示「AI 请求失败 / 仅证据检索」**：先确认 `.env` 里 `AI_API_KEY` 是真实有效的 key（找管理员要）。若已配置仍失败，看后端启动终端里的日志：会打印 `Site assistant AI request failed (model=xxx): HTTP xxx ... body=...`，据此判断是 key 无效、模型名不对还是网关不通。
- **接口 401**：见第 4 节（密码坑）。

---

## 4. ⚠️ 最容易踩的坑：密码导致智能客服 401

后端对 `/api/ai`、`/api/content`、`/api/research` 加了密码中间件（`PasswordMiddleware`），规则是：

- `READ_PASSWORD` 和 `EDIT_PASSWORD` **都为空** → 完全放行，所有接口可用（**本地自测推荐这样**）。
- 只要配置了其中任意一个 → 请求头必须带正确的 `x-access-password`，否则返回 **401**。

而前端的密码是在点「进入编辑」并输入 `EDIT_PASSWORD` 成功后，才写入 `sessionStorage`、后续请求才会带上这个头。

**结论 / 避坑建议**：

- 本地自己调试：`.env` 里 `READ_PASSWORD` 和 `EDIT_PASSWORD` 都**留空**，智能客服和读接口直接就能用，省事。
- 如果你配置了 `READ_PASSWORD` 却没配 `EDIT_PASSWORD`（或没点「进入编辑」解锁），那么智能客服的 `/api/ai/ask-site` 会 401 → 前端表现为 AI 不可用。

## 5. 实时修改（编辑模式 + 持久化）配置

### 依赖的环境变量

| 变量 | 作用 | 不配置时 |
|------|------|----------|
| `DATABASE_URL` | Postgres 连接串，内容/画像/用户档案都存这里 | 页面能看（默认数据），但**保存失败（503）**，刷新后改动丢失 |
| `EDIT_PASSWORD` | 解锁编辑 + 写接口鉴权 | 视为「无需编辑密码」，任意密码都能解锁、写接口放行 |
| `READ_PASSWORD` | 保护读接口 | 留空则读接口完全开放 |

### 操作流程

1. 顶栏点击 **「进入编辑」**（`EditorModeButton`）。
2. 输入 `EDIT_PASSWORD`（若 `.env` 里没配，随便输入即可通过）。
3. 解锁后，支持编辑的页面（核心结论、调研结论、下一步建议、家庭包结论/洞察、竞品分析、画像/用户等）会出现**铅笔按钮**。
4. 修改内容 → 保存 → 期望 toast **「保存成功」**；刷新页面改动仍在，其他人 30s 内也会看到（轮询同步）。

### 本地准备一个 Postgres

任意可连接的 Postgres 即可，表会自动建。用 Docker 最快：

```bash
docker run --name insighthub-pg -e POSTGRES_PASSWORD=postgres -p 5432:5432 -d postgres:16
```

对应 `.env`：

```bash
DATABASE_URL=postgresql://postgres:postgres@localhost:5432/postgres
```

### 验证是否成功

- 保存后 toast「保存成功」，刷新仍保留 → 成功。
- toast「保存失败，数据库可能未配置」→ `DATABASE_URL` 没配或连不上；看后端终端是否有 `DATABASE_URL not set — content API will return 503` 的 warn。

### 存储说明（了解即可）

- 通用内容存 `content_store` 表（key-value + jsonb），由 `useContentStore` 走 `GET/PUT /api/content/:key`。
- 画像/用户档案存 `research_*` 表，走 `/api/research/*`。
- 两者都在服务启动时自动建表，无需手动 migration。

## 6. 运行方式（开发 vs 生产）

| 模式 | 命令 | 访问地址 | 说明 |
|------|------|----------|------|
| 开发 | `npm run dev` | http://localhost:5173 | 前端 Vite(5173) + 后端 Nest(3000)；Vite 把 `/api` 代理到 3000 |
| 生产 | `npm run build` 然后 `npm start` | http://localhost:3000 | 单端口，Nest 同时托管前端静态资源 |

> 前端所有接口都用**相对路径** `/api/...`，没有单独的 `VITE_API_URL`。
> - 开发一定要用 `npm run dev`（或至少同时启动前端和后端），只启动其中一个会导致 `/api` 请求失败。
> - 生产模式若没先 `npm run build` 就 `npm start`，会因为缺少 `dist/` 而起不来。

## 8. 快速排错对照表

| 现象 | 最可能原因 | 处理 |
|------|-----------|------|
| 智能客服只给证据、无 AI 总结 | `AI_API_KEY` 未配/无效 | 填真实 key；看后端日志 `Site assistant AI request failed` |
| 智能客服接口 401 | 配了 `READ_PASSWORD` 但没解锁编辑 | 本地把密码留空，或先「进入编辑」输入 `EDIT_PASSWORD` |
| 换了模型后 AI 请求失败 | 推理模型不接受某些参数 | 代码已自动兼容；仍失败则看日志 `body=...` |
| 保存提示「保存失败，数据库可能未配置」 | `DATABASE_URL` 未配/连不上 | 配置并确认 Postgres 可连 |
| 页面能看但改了刷新就没了 | 同上（没连数据库） | 同上 |
| 看不到铅笔按钮 | 没「进入编辑」解锁 | 顶栏点「进入编辑」 |
| `/api/...` 全部请求失败 | 只启动了前端或只启动了后端 | 用 `npm run dev` 同时起两端 |
| 生产 `npm start` 起不来 | 没先 `npm run build` | 先 build 再 start |
